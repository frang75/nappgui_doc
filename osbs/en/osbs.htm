langrev. h1.Osbs ep.There is no neat distinction between operating system software and the software that runs on top of it. Jim Allchin p.osbs (Operating System Basic Services) is a portable wrapper that allows applications to communicate with the operating system core at the level of processes, memory, files, networks, clocks, etc. This communication is carried out through a series of system calls system_calls which vary according to the operating system for which we are programming. It is the non-graphic lowest level API to communicate with hardware devices and access the machine resources. Below are the device drivers managed directly by the kernel, to which applications have access denied. img(system_calls.svg,600,.8).System calls are the gateway to the operating system kernel. Schema showing how the osbs library connects with the hardware through system calls. p.Darwin, the macOS kernel, and Linux are Unix-like systems, therefore, they share the same system calls (with subtle differences). But Windows presents a radically different architecture and function set. The NAppGUI osbs library is nothing more than a small wrapper that internally handles these differences and provides a common way to access the same resources on different platforms osbs_depends. Osbs library dependency tree. h2.Standard C library p.The C standard library (cstdlib) is not part of the C language, but implements functions of great utility for the developer that solve typical programming problems. Any C programmer has used it more or less and its study is usually linked to learning the language itself cstd_book. img(cstd_book.png,420,.3).A complete reference to the C library is found in the P.J.Plauger book. "The C Standard Library" book cover. p.This library is located halfway between the application and system calls and provides a portable API for file access, dynamic memory, I/O, time, etc clib_depends. It also implements mathematical functions, conversion, search, string management, etc. In one way or another, NAppGUI integrates its functionality into osbs, so it's not necessary (or advisable) to use cstdlib directly in the application layer. The reasons that have motivated this design decision can be summarized in: img(clib_depends.svg,320,.4).The functionality of the C library has been integrated in osbs, avoiding its direct use in applications. Schema showing the osbs library in relation to the standard C library. li.Small differences: Unix-like systems do not support the secure cstdlib versions implemented by Microsoft (strcpy_s() and others). The use of classical functions (without the suffix _s) is insecure and will trigger annoying warnings in Visual Studio. li.Security: Related to the previous one, osbs avoids buffer overflow vulnerabilities in the processing of memory blocks and strings. li.Duplicity: Much of the functionality of cstdlib is already implemented in osbs using direct system calls (files, dynamic memory, I/O, time, etc.) li.Completeness: The cstdlib functions related to files (fopen() and others) do not include support for directory management. bfile presents a complete API based on system calls. li.Performance: In certain cases, especially in mathematical functions and memory management, it may be interesting to change the implementation of cstdlib to an improved one. All applications will benefit from the change, without having to modify your code. li.Clarity: The behavior of some cstdlib functions is not entirely clear and can lead to confusion. For example, strtoul has a very particular functioning that we must remember every time we use it. code. char *s1 = "-56"; char *s2 = "asCr"; char *s3 = "467Xd"; int v1, v2, v3; v1 = strtoul(s1, NULL, 10); // v1 = 4294967240, errno = OK v2 = strtoul(s2, NULL, 10); // v2 = 0, errno = OK v3 = strtoul(s3, NULL, 10); // v3 = 467, errno = OK code. li.Style: The use of osbs functions does not break the aesthetics of an application written with NAppGUI. code. real32_t a1 = 1.43f; real64_t a2 = .38; real32_t c = (real32_t)cosf((float)a1); real64_t t = (real64_t)tan((double)a2); ... real32_t c = bmath_cosf(a1); real64_t t = bmath_tand(a2); code. li.Independence: NAppGUI internally uses a very small subset of cstdlib functions.
