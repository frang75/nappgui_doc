h1.Gui p.La librería Gui permite crear interfaces gráficas de usuario de forma sencilla e intuitiva. Solo está disponible para aplicaciones de escritorio por razones obvias gui_depends, al contrario que el resto de librerías que pueden utilizarse también en aplicaciones por línea de comandos. img(gui_depends.svg,300,.4).Dependencias de Gui. Ver sdk. Árbol de dependencias de la librería gui. p.Al igual que draw2d y osbs Gui se apoya en las APIs de cada sistema operativo gui_tech. Además de las ventajas ya comentadas en estos dos casos, el acceso nativo a elementos de interfaz provocará que nuestros programas estén totalmente integrados en el escritorio y acordes al tema visual presente en cada máquina native_look_and_feel. img(gui_tech.svg,400,.5).Gui conecta con las APIs de interfaz de cada sistema operativo. Dependencias de gui en función de cada sistema operativo. img(native_look_and_feel.svg,700,.75).Las interfaces creadas con Gui se adaptarán al estilo de cada entorno de ventanas. Imagen de un mismo programa corriendo en Windows, macOS y Linux. h2.Composición declarativa p.La librería Gui se aleja del concepto de tratar las ventanas (o cuadros de diálogo) como un recurso externo del programa. Por el contrario, estas se crean directamente desde el código fuente evitando maquetarlas mediante editores visuales visual_editor. Debemos tener en cuenta que los gestores de ventanas utilizan diferentes tipografías y plantillas, por lo que especificar posiciones y tamaños concretos para los elementos no será portable entre plataformas control_fixed_size. Por el contrario, en Gui los controles se ubican en una rejilla virtual denominada Layout, que calculará su ubicación y tamaño final en tiempo de ejecución y en función de la plataforma layout_compose. img(visual_editor.svg,580,.65).Los editores de recursos no son buenos aliados para crear complejas interfaces dinámicas. Menos aún si queremos portarlas entre plataformas. Captura del editor de interfaces incluido en Xcode. img(control_fixed_size.svg,460,.6).Utilizar medidas fijas para los controles no se adaptará bien al migrar el programa. Controles Label y Button en Windows, macOS y Linux. img(layout_compose.svg,460,.6).El Layout calcula la posición y tamaño de los componentes en tiempo de ejecución. Controles Label y Button en Windows, macOS y Linux. p.Además, otro hecho relevante es que las interfaces son objetos vivos sujetos a cambios constantes. Un claro ejemplo son las traducciones, que alteran la ubicación de los elementos debido a la nueva dimensión del texto translate. Gui se adaptará a estos eventos de forma automática, recalculando posiciones para mantener una maquetación coherente. img(translate.gif,0,.65).Las ventanas se adaptan automáticamente a cambios en tiempo de ejecución. Los controles son los elementos finales con los que interactúa el usuario para introducir datos o lanzar acciones. Las vistas son regiones rectangulares de tamaño relativamente grande donde se representa información mediante texto y gráficos, pudiendo responder a los eventos de teclado o ratón. Por último, todos estos elementos se agruparán en paneles y se maquetarán mediante layouts. img(window_anatomy.svg,600,.65).Partes destacables en una ventana de interfaz. Ventana de interfaz donde se resaltan las partes más importantes. li.label. Pequeños bloques de texto descriptivo. li.button. Botones de pulsación, casillas de verificación o botones de radio. li.popup. Botón con lista desplegable. li.edit. Cuadro de edición de texto. li.combo. Cada uno de los elementos del menú. h2.Eventos GUI p.Las aplicaciones de escritorio están continuamente esperando a que el usuario realice alguna acción sobre la interfaz: Pulsar un botón, arrastrar un slider, escribir un texto, etc. Cuando esto ocurre, el gestor de ventanas detecta el evento y lo notifica a la aplicación gui_event, la cual debe proveer un manejador del evento con el código a ejecutar. Por ejemplo, en eventhandler definimos un manejador para responder a la pulsación de un botón. Evidentemente, si no hay manejador asociado, la aplicación ignorará el evento. li.Utiliza event_params para obtener los parámetros asociados al evento. Cada tipo de evento tiene sus propios parámetros. Ver allevents. li.Utiliza event_result para escribir la respuesta al evento. Muy pocos eventos requieren enviar una respuesta. img(gui_event.svg,460,.6).Notificación de un evento a través del manejador.
