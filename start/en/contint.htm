langrev. h1.Continuous integration ep.A Continuous Integration system will compile projects continuously automatically and on all platforms, as we make changes to the repository. p.We already saw in win_mac_linux how to move our applications between different platforms and development environments. This is a great advantage over traditional native programming, since we can generate optimized executables for each CPU and operating system without having to modify a single line of the source code. Now we are going to go one step further by automating the compilation and test phases. p.An average work team can modify the code of a project several times a day. In cross-platform environments, such changes must be compiled and tested on several machines as soon as possible in order to correct errors or anomalies. For this, the Continuous Integration (CI) system will work continuously, in the background and autonomously each time the content of the repository is modified contint. img(contint.svg,640,.75).Continuous Integration System will work autonomously in the background. Basic outline of a system of continuous integration. h2.Build grid p.The first thing we have to do is set up a heterogeneous computer network where different machines coexist with the appropriate compilers buildgrid and that can communicate with each other ssh. The main actors of this network are: li.Build Grid: Network formed by all the machines that will cooperate in compilation tasks. They can be distributed over the Internet. li.Build Master: Computer within the network that runs the Continuous Integration utility, coordinates the rest of the machines, stores the results and reports errors. li.Build Host/Slave: Machine for compilation and test tasks, which will be remotely managed by the Build Master. li.Build Agent: Configuration that identifies a compiler and CPU version. For example, the agent v142_x64 represents Visual Studio 2019 (VC++) 64bit. An agent can be executed by any Host that has the appropriate compiler installed. The Build Master will be in charge of balancing the load. li.Build Profile: Profile that indicates all targets (libraries and/or executables) to be generated within a solution and which agents must intervene. The Build Master can handle several profiles at the same time. li.Build Config: Indicates the level of optimization and debug options of the compilation: Debug, Release, ReleaseWithAssert. Configurations. img(buildgrid.svg,640,.75).Example Build Grid. Scheme of a network of compilers, governed by a Build Master. h2.Command line p.However complex the Build Profile may be, the remote compilation process can be divided into four steps: li.Update: The working copy of the host, if outdated. li.CMake: To generate compilation solutions .sln, .xcodeprojor Makefile. li.Build: Compile by msbuild, xcodebuild or make depending on the platform. li.Copiar: Compilation results, errors, warnings, libraries or executables. The Master will package all products and submit reports. p.El Build Master you must use the command line version of all the tools involved and launch them on the remote host using SSH. The warnings and errors found in the code will keep them in the archives warnings.txt and errors.txt respectively. h3.Xcode from command line p.Like Visual Studio, Xcode can be executed from the command line. Unlike MSBuild the notices and errors write them in stderr, preceded by tags warning: and error: respectively. h3.GCC/Make from the command line p.In Linux the project is generated by files Makefile, which contain compilation scripts based on GCC.
