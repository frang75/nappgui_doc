langrev. h1.Resources p.Resources are data necessary for the application but do not reside in the executable area. In other words, they are not directly accessible through the program variables, but must be preloaded to be able to use them. The most common are the texts and the images used in the user interface, although any type of file can become a resource (sounds, fonts, 3d models, html pages, etc). We will continue working on the application die die_application, this time deepening the use of resources. img(die_application.png,0,.65).Die application. Capture the sample application Die. h2.Resource types li.Texts: Although it is very easy to include texts in the code as variables of C, in practice this is not advisable for two reasons: The first is that, normally, it is not the programmers who write the messages that the program shows. By separating them into a separate file, other team members can review and edit them without having to access the code directly. The second reason is internationalization. It is an almost indispensable requirement today to be able to change the language of the program and this may involve several members of the team, as well as the fact that several text strings will refer to the same message. Therefore, extracting them from the source code will be almost indispensable. li.Images: It is not usual for the icons of the program to change according to the language, although this may be the case. The tricky thing here is to transform a .jpg or .png file into a C variable serialimg. You have to serialize the file and paste it into the code, something very tedious and difficult for the programmer to maintain. It is preferable to have the images in a separate folder and access them at runtime. code(cpp,serialimg)..png image embedded in the source code. const uint32_t IMG_SIZE = 1262; const byte_t IMG[] = { 0x89, 0x50, 0x4E, 0x47, 0x0D, 0x0A, 0x1A, 0x0A, 0x00, 0x00, 0x00, 0x0D, 0x49, 0x48, 0x44, 0x52, ... }; code. li.Files: Apart from text and images, any file can become a resource. In this case, the application will receive a block of bytes with the contents thereof, which must know how to interpret. h2.Create resources p.If we go to the source directory of the application (C:\nappgui\die), we see that there is a folder called res added by CMake when creating the project. CMake will warn if there are untranslated texts Resource warnings. li.Those resources only present in localized folders will be ignored. It is imperative that the global version exists. li.No "subpackets" of resources are allowed. Only two levels will be processed: /res/pack for the global and /res/pack/local for those localized. li.The existing resources in the root folder (/res) will be ignored. All resources must be linked to a package. li.The localized texts must have the same identifier as their global corresponding. Otherwise, different messages are considered. li.To create the localized version of an image or another file, include it in its corresponding localized folder (eg /es/all/es_es/cards.png) using the same file name as the global version. li.To name the localized folders, use the two-letter language code https://en.wikipedia.org/wiki/List_of_ISO_639-1_codes''ISO 639-1 (en, es, fr, de, zh, ...) and, optionally, the two-letter country code https://en.wikipedia.org/wiki/ISO_3166-1''ISO-3166 (en_us, en_gb, ...). h2.Loading and translation p.For each resource package, CMake has created a *.h with the same name as the folder: all.h in this case reshead. When establishing NRC_EMBEDDED in the command desktopApp(), we tell CMake to process the resources of the Die project. We can also choose the option NRC_PACKED of which we will speak next. When we press [Generate] CMake traverses the subfolders within the directory res of each project, calling the nrc utility (NAppGUI Resource Compiler) resources. This program is in the folder prj/scripts of the distribution of the SDK. For each resource package, nrc creates two source files (one .c and a .h) and links them to the project. The .h contains the identifiers and the constructor that we have seen in reshead. For its part, the .c performs the package implementation based on the content of each folder and the nrcMode modality. img(resources.svg,-1,.75).Processing resources using CMake and nrc. Schema showing how the nrc compiler processes the resource folders. bq.Files created by nrc are considered generated code and are not saved in the folder src but in the folder build. They will be updated every time you press [Generate] in CMake, regardless of the platform on which we are working. On the contrary, the original resource files (located in the folder res) are considered part of the source code. h2.Resources distribution p.In the previous chapter, when creating the Visual Studio solution, we indicated that we had to use the constant NRC_EMBEDDED in the sentence desktopApp() inside the file CMakeLists.txt. There are two other modalities more related to the management of resources and that can be configured separately within each command desktopApp(): li.NRC_NONE: CMake will ignore the contents of the folder res, except for the icon of the application. No resource packages will be generated even if there is content within this folder. li.NRC_EMBEDDED: The resources, with all their translations, are integrated as part of the executable embedded_resources. It is a very interesting option for applications of small or medium size, since in a single file *.exe we will supply the entire program. You will not need an installer and we will have the certainty that the software will not fail due to the lack of an external file. The downside is that, obviously, the size of the executable will grow considerably so it is not advisable in programs with many resources, very heavy, or with a multitude of translations. li.NRC_PACKED: For each resource package, a *.res file will be created external to the executable that will be loaded and released at runtime as needed packed_resources. The advantages of this method are the disadvantages of the previous one and vice versa: Smaller executables, but with external dependencies (the own ones .res) that must be distributed together. The use of memory will also be optimized, being able to load the *.res on demand. img(embedded_resources.png, 560, .85).Distribution of a macOS application with embedded resources. Capture of the macOS distribution with embedded resources. img(packed_resources.png, 560, .85).Distribution of the same macOS application with packaged resources. Capture of the macOS distribution with packaged resources. p.CMake manages the location of the resource packages for us. In Windows and Linux applications, it will copy all *.res in the executable directory. In macOS it will place them in the folder resources of the bundle. A very important fact is that we do not have to modify the source code when going from one modality to another. nrc is already responsible for managing the load according to the type of package. Something logical can be to start with NRC_EMBEDDED, and if the project grows, change to NRC_PACKED. We just have to press again [Generate] (as we always do) and recompile the project so that the change becomes effective. bq.In Windows and Linux files *.res they must always be installed in the same directory as the executable. In the case of macOS, CMake generates a bundle ready for distribution and installs the resource packages in the directory /resources of said bundle. h2.Resource warnings p.nrc is a silent script whose work is integrated into CMake's build process, going unnoticed in most cases. But there are times that it detects anomalies in the resource directories and it must inform us in some way. In these cases, a red line will appear in the CMake console indicating the project and affected package(s) nrc_warnings. The details dumps in the file NRCLog.txt located in the generated resources folder (CMake shows the complete path). img(nrc_warnings.png, 0, 1).nrc has encountered anomalies when processing resources. This image will be "embedded" in the executable and the operating system will use it to represent the application on the desktop app_icons. Windows and Linux also use it in the title bar of the window. We have three versions: li.logo256.ico: Version for Windows Vista and later. They should include the resolutions: 256x256, 48x48, 32x32 and 16x16. li.logo48.ico: Version for Linux and Visual Studio 2008 and 2005, which do not support resolutions of 256x256. This version only includes: 48x48, 32x32 and 16x16. li.logo.icns: Version for macOS. Resolutions 512x512, 256x256, 128x128, 32x32 and 16x16 both in normal resolution (@1x) and Retina Display (@2x). img(app_icons.png,0,.8).Application icons in the Windows taskbar. To change the default icon, open the files logo* with some graphic editor icon_editor, make the changes and press again [Generate] in CMake.
